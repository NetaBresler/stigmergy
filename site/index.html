The bug in how we build agent teams

CrewAI. AutoGen. OpenAI Swarm. Claude Agent Teams. Almost every multi-agent framework shipping today encodes the same assumption: if you want many agents to work together, give them a manager.

This isn't architecture. It's cargo-culting the org chart — we saw how humans coordinate and made the LLMs wear suits.

The pattern plateaus for a predictable reason. The manager's context window is the bottleneck. Every decision passes through it. Every worker's output is re-read and re-summarized. Token cost grows superlinearly. Latency compounds. The manager's attention is the ceiling on the team's intelligence.

There is another way

In 1959, the French biologist Pierre-Paul Grassé watched termites rebuild a damaged nest and noticed they weren't talking to each other. They were coordinating with the nest. Each deposit of pheromone-laced mud changed the environment in a way that suggested the next deposit to whoever showed up next.

He called it stigmergy — from Greek stigma (mark) and ergon (work). Work that guides work.

Agents read from a database, deposit signals as they work, and the next agent picks up whatever the medium suggests it should do next. Stale signals decay automatically, so the colony forgets plans that aren't working. Specialization emerges from the pressure landscape instead of being assigned by a planner.

Why this beats hierarchy

01

Coordination cost stops scaling with team size

Adding a tenth agent means one more reader of the medium. Linear scaling instead of quadratic.

02

No single point of failure

Kill a worker and its unfinished work stays visible. The next agent picks it up. Crashes are recoverable by default.

03

Specialization emerges

Agents self-select work based on what the medium needs — not on what a planner decides from a stale context window.

04

Forgetting is built in

Pheromones evaporate. Stale plans die automatically. You don't need a "reconsider the strategy" step.

05

The medium holds more than a context window

A Postgres table can hold millions of signals. No agent ever needs to see all of them at once.

And the catch

If the task is single-threaded reasoning, stigmergy adds nothing. It pays off when there's genuine parallelism.

The six primitives

A system is stigmergic if and only if it has all six. Miss one and you have something else — a knowledge base, a queue, a workflow engine, or a manager-in-disguise.

1

Medium

The one place agents read from and write to. The only legitimate coordination channel. Reference implementation: Postgres.

Anti-pattern: a message bus. Signals flow through instead of being deposited — nothing to decay, nothing to persist.

2

Decay

Every signal evaporates unless reinforced. Three supported mechanisms: explicit expiry, strength decay, reinforcement-only. The framework refuses to register a signal type without one.

Anti-pattern: "we'll add decay later." Stigmergy without decay is just a shared database.

3

Role specialization

A role bundles: what signals the agent reads, what it writes, what its local query looks like. Roles don't reference each other — only signals.

Anti-pattern: a Supervisor role that reads everything. That's a manager. Delete it.

4

Agent

The identity that enacts roles — distinct from the role it's enacting. Stable id, persistent identity documents (soul, skills, memory), selects which role to play each tick. Polyethism.

Anti-pattern: agents messaging each other directly. Coordination is mediated by signals. Always.

5

Locality

Every role declares a local query — the slice of the medium it can read. The framework enforces it. You cannot micro-manage what you cannot see.

Anti-pattern: "the Worker also needs to see the Validator queue." No. If it needs info, a signal carries it.

6

Validated reinforcement

Not every trace reinforces equally. Reinforcement is a distinct operation from deposition, and changes strength / expiry / visibility — never content. Rule-based, agent-based, or human-in-the-loop.

Anti-pattern: every signal starts at full strength. Whoever runs first wins, regardless of quality.

How this differs from what's shipping

LangGraph / CrewAI / AutoGen Claude Agent Teams Stigmergy
Coordinator Graph / crew lead / conversation manager Team-lead session None. The medium coordinates.
Scaling cost Super-linear in team size Bounded by lead's context Linear — each agent adds one reader
Failure mode Manager crash halts team Lead crash halts team Worker crashes are recoverable
Memory In-context, re-serialized per handoff In-context, per session In the medium. Inspectable by SQL.
Forgetting Explicit (prune / re-plan) Explicit Built in. Decay is a primitive.
Specialization Assigned by planner Assigned by lead Emerges from pressure

All of those are perfectly good for small, single-threaded teams. Stigmergy is the alternative for when your colony has real parallelism.

Get started in three commands

Node 22+. Postgres is the default substrate; PGlite works in-process for tests and demos, no server required.

1

Install

Stigmergy isn't on npm yet — it installs directly from GitHub while we wait for its first real user (Phase 2).

npm install github:NetaBresler/stigmergy
2

Run the worked example

A three-agent colony — Scout, Worker, Validator — that deposits signals, competes for claims, and reinforces the winning pheromones while the weak ones decay.

git clone https://github.com/NetaBresler/stigmergy
cd stigmergy
npm install
npx tsx examples/niche-discovery.ts
3

Define your own colony

Six functions. Everything else is opinion.

import {
  defineMedium, defineSignal, defineRole,
  defineAgent, defineValidator, run,
} from "stigmergy";

const medium = defineMedium({ url: process.env.DATABASE_URL });

defineSignal({
  type: "candidate",
  decay: { kind: "strength", halfLifeHours: 6 },
  shape: { body: "text", priority: "number" },
});

defineRole({
  name: "worker",
  reads:  ["candidate"],
  writes: ["result"],
  localQuery: { type: "candidate", orderBy: "strength", limit: 1 },
});

run(agent, async (ctx) => {
  const [candidate] = await ctx.read("candidate");
  if (!candidate) return;
  await ctx.deposit("result", { body: await process(candidate) });
});

This is not a knowledge base

The sharpest question about stigmergy, and the one most people get wrong the first time.

Dimension Knowledge base (RAG) Stigmergic medium
Who writesHumans curate; batchEvery agent, as it works
Time dimensionStatic until re-indexedDecays; recency encodes priority
What it encodesFactsIntent, attention, claim-state
Agent's question"What do I need to know?""What does the colony need next?"
Failure modeStale facts misleadStale signals decay away

A knowledge base tells an agent what is true. A medium tells an agent what to do. You want both. They are not substitutes.
